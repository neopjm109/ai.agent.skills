# Goal

Produce the design-time API specification that exposes domain capabilities over RESTful
interfaces: resources, endpoints, request/response DTO contracts, routing/versioning,
authentication/authorization, and standardized error models (an OpenAPI-style design
document). This is a **design-time skill — it does not generate code**. It emits an
`api_specification` artifact; the **runtime API code (Controllers, DTO classes, mapping,
routing) is generated by backend/api-generator**. See INVENTORY.md for the
api-spec-generator ↔ api-generator boundary.

# Inputs

```yaml
domain_model:
  aggregates:
    - name: Order
      root: Order
      entities: [Order, OrderItem]
  domain_services:
    - name: PricingService
      responsibility: compute order total
  domain_events:
    - name: OrderPlaced
unified_requirements:
  functional_requirements:
    - id: FR-1
      description: Customers create and view orders
  non_functional_requirements:
    - id: NFR-1
      description: All write endpoints require authentication
  business_rules:
    - id: BR-1
      description: An order cannot be created with zero items
```

# Output

```yaml
api_specification:
  base_path: /api
  version: v1
  resources:
    - name: orders
      aggregate: Order
  endpoints:
    - { method: POST, path: /api/orders, operation: createOrder, refs: [FR-1] }
    - { method: GET,  path: /api/orders/{id}, operation: getOrder, refs: [FR-1] }
  request_dtos:
    - name: CreateOrderRequest
      fields: [customerId, items[]]
      validation: items must be non-empty        # refs: BR-1
  response_dtos:
    - name: OrderResponse
      fields: [id, status, total, placedAt]
  routing:
    versioning: URI path (/api, v1)
  security:
    authentication: JWT bearer
    authorization:
      - { endpoint: POST /api/orders, roles: [CUSTOMER] }   # refs: NFR-1
  error_models:
    - { code: VALIDATION_ERROR, http: 400 }
    - { code: NOT_FOUND, http: 404 }
    - { code: UNAUTHORIZED, http: 401 }
    - { code: INTERNAL_ERROR, http: 500 }
```

# Workflow

## Step 1 — Map domain to API resources

Convert aggregates to REST resources, domain services to operations/endpoints, and domain
events to async or webhook endpoints where required by requirements.

## Step 2 — Define endpoints

Design RESTful endpoints (GET/POST/PUT/DELETE) per resource with paths, operation names,
and success status codes, tracing each to a functional requirement.

## Step 3 — Define DTO contracts

Specify request and response DTO contracts with field-level validation rules. DTOs must
never expose domain entities directly. These are contracts only — the concrete classes are
built by backend/api-generator.

For **create/update** operations, the request DTO must be *field-complete* against the
target aggregate: expose every field the client must supply to satisfy the aggregate's
invariants, **expanding value objects into their component fields** (e.g. a `Money` value
object → `amount` **and** `currency`; an `Address` VO → its line/city/postal fields). A
required field may be omitted from the request DTO only when the domain model marks it as
server-assigned/derived/defaulted (e.g. `status` defaults to `PENDING`, `id`/`placedAt`
server-generated). Never silently drop a client-supplied required field — that yields a
create endpoint that cannot populate the aggregate.

## Step 4 — Define routing, security, and error models

Define base path and versioning strategy, authentication method and authorization rules
(traced to non-functional requirements), and a standardized set of error models.

# Rules

- Never generate implementation code — emit a design spec only. Runtime Controllers/DTOs
  are produced by backend/api-generator (do not duplicate its work here).
- Thin-API principle: no business logic in the spec's endpoints — only orchestration and
  mapping responsibilities are described.
- DTO contracts must never leak domain internals; no entity exposure.
- Create/update request DTOs must cover every client-supplied required field of the target
  aggregate, with value-object fields expanded into their components; omit a required field
  only if the domain marks it server-assigned/derived/defaulted. (Verified by
  `blueprint-validator`'s write-DTO completeness check.)
- Use resource-based (noun) naming and a consistent base path; avoid verb-based endpoints.
- Every API element must reference a domain element, requirement, or business rule
  (traceability).
- The spec is complete only when all aggregates are exposed, endpoints and DTO contracts
  are defined, and security and error models are standardized.

# Examples

Input:

```yaml
domain_model:
  aggregates:
    - name: User
      root: User
unified_requirements:
  functional_requirements:
    - id: FR-1
      description: Register a new user
  non_functional_requirements:
    - id: NFR-1
      description: Registration endpoint is public; profile read requires auth
```

Output (abridged):

```yaml
api_specification:
  base_path: /api
  version: v1
  resources:
    - { name: users, aggregate: User }
  endpoints:
    - { method: POST, path: /api/users, operation: registerUser, refs: [FR-1] }
    - { method: GET,  path: /api/users/{id}, operation: getUser }
  request_dtos:
    - name: RegisterUserRequest
      fields: [email, displayName, password]
      validation: email format required; password min length 8
  response_dtos:
    - name: UserResponse
      fields: [id, email, displayName]     # password never returned
  security:
    authentication: JWT bearer
    authorization:
      - { endpoint: POST /api/users, roles: [PUBLIC] }
      - { endpoint: GET /api/users/{id}, roles: [USER] }   # refs: NFR-1
  error_models:
    - { code: VALIDATION_ERROR, http: 400 }
    - { code: NOT_FOUND, http: 404 }
# NOTE: runtime UserController + DTO classes are generated by backend/api-generator.
```
